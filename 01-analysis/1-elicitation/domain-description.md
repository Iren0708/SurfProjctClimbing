# Описание домена и ограничения

> Этап 1. Артефакт 1 проекта: согласованное описание домена + список ограничений.
> Заполняется после ответов заказчика ([customer-questions.md](customer-questions.md)).

## Описание домена

**Бизнес.** «Вертикаль» — небольшой скалодром в бывшем складском ангаре, работающий
третий сезон. Клуб организует **групповые тренировки** по скалолазанию.
Тренировка длится 1.5 часа и проходит в одном из **форматов**: болдеринг с инструктажем
для новичков или трассы с верёвкой для опытных. Каждую тренировку ведёт **инструктор**
(сейчас их пятеро, в сезон нанимаются подработчики).

**Процесс.** Владелец (Оля) заранее формирует **расписание на неделю вперёд** через существующую
админку: для каждой тренировки задаются дата и время старта, зона/формат, назначенный инструктор
и число мест. Инструкторы расписание не редактируют. На каждую тренировку — **один инструктор**.
Лимит мест в слоте = `min(потолок типа тренировки, свободные места в слоте)`: для новичковых
групп потолок 8, для опытных — до 16. Клиент видит свободные **слоты** (время, зона/формат,
инструктор, всего/свободно мест, **цена**) и **записывается** на одно место, указывая, придёт
он со своим снаряжением или возьмёт снаряжение **напрокат**. Лимиты мест в группе и прокатного
фонда проверяются **раздельно**: своё снаряжение не занимает прокатный фонд, но место в группе
занимает. Каждый участник записывается **отдельно** со своим аккаунтом (имя + телефон) —
групповая бронь одним клиентом не поддерживается. До тренировки клиент может **отменить**
запись: при отмене за 2 часа и раньше место освобождается; поздняя отмена (<2 ч) разрешена,
но фиксируется без штрафов — при повторных неявках админ решает вручную. Сейчас запись клиентов
ведётся вручную (Telegram + тетрадь), что приводит к двойным записям и путанице, особенно
в часы пик.

**Цель клиентского приложения.** Дать клиентам самостоятельную онлайн-запись на тренировки
(просмотр и фильтрация слотов, бронирование одного места, выбор снаряжения, отмена, свои
бронирования, push-напоминания), исключив ручную путаницу и двойные брони. Управление
расписанием, отметка оплаты, явки/неявки и работу инструкторов/владельца обеспечивает
**существующая инфраструктура**; приложение потребляет её данные через API.

**Ключевые объекты домена:**

- **Тренировка / слот** — конкретное занятие группы: дата/время старта, зона/формат, инструктор, всего и свободно мест, цена. Создаётся в существующей инфраструктуре; клиенту доступен только для чтения.
- **Зона / Формат** — вариант тренировки (болдеринг для новичков / трассы для опытных), определяет длительность и потолок мест (справочные данные, read-only).
- **Инструктор** — ведёт тренировку; для клиента — справочные данные слота (read-only).
- **Снаряжение** — скальники, страховочная система; может выдаваться напрокат (лимит прокатного фонда учитывается отдельно от мест в группе).
- **Запись (бронь)** — резерв **одного места** клиентом на конкретный слот: своё/прокатное снаряжение, статус. Одна запись = один клиент.
- **Клиент** — регистрируется (имя + телефон), записывается на тренировки, отменяет запись, получает напоминания.

## Пользователи и роли

> **Скоуп текущей работы — только клиентское мобильное приложение и API для него.**
> Роли Инструктор и Владелец/Администратор обслуживаются **существующей инфраструктурой**
> (бэкенд, расписание, админка) и в этот скоуп не входят.

| Роль | Описание | Цели | В скоупе |
| :-- | :-- | :-- | :-- |
| **Клиент** | Гость клуба, занимается скалолазанием. Пользуется с телефона. | Увидеть и отфильтровать свободные слоты (с ценой), записаться на одно место, выбрать своё/прокатное снаряжение, видеть свои бронирования, отменить запись, получать push-напоминания (за 24 ч и 2 ч до старта). | **Да** |

> Инструктор и Владелец/Администратор взаимодействуют с системой через **уже реализованные**
> интерфейсы существующей инфраструктуры. Инструктор видит свои тренировки, отмечает явку/неявку,
> может отменить тренировку с причиной. Владелец управляет расписанием, слотами, инструкторами,
> записями и отметкой оплаты. Клиентское приложение лишь потребляет подготовленные ими данные
> (слоты, зоны/форматы, инструкторы) через API.

## Ограничения

**Бизнес-ограничения**

- **Лимит мест слота** = `min(потолок типа тренировки, свободные места в слоте)`. Для новичковых групп потолок 8, для опытных — до 16. Один инструктор на группу.
- **Лимит прокатного снаряжения** учитывается отдельно от мест в группе; своё снаряжение прокатный фонд не занимает.
- **Одна запись = один клиент** — групповая бронь (запись друзей/семьи одним аккаунтом) не поддерживается.
- **Политика отмен:** бесплатная отмена не позднее чем за **2 часа** до старта — место освобождается;
  поздняя отмена (<2 ч) разрешена, фиксируется без денежных штрафов; при повторных неявках
  админ видит историю и решает вручную.
- **Оплата офлайн** (наличные / перевод на карту на месте) — приложение показывает цену и
  фиксирует бронь. Статус «оплачено/нет» проставляет админ в существующей инфраструктуре.
- **Период списка слотов по умолчанию — 7 дней (R-027).** По умолчанию клиенту показываются
  слоты на **ближайшие 7 дней** от текущего момента — это совпадает с горизонтом планирования
  расписания («на неделю вперёд»). Более длинный период доступен через **фильтр дат**. При
  отсутствии расписания на ближайшие дни показывается **empty state** («Пока нет доступных
  тренировок»).
- **Отмена тренировки скалодромом / по профилактике (R-008).** Если тренировку отменяет клуб (организатор или
  из-за профилактики/проверки снаряжения), бронь на стороне клиента **не удаляется**, а получает статус **«Отменена
  скалодромом»** с указанием причины; клиенту приходит **push-уведомление**. **Повторная запись на
  отменённый слот запрещена.** Само действие отмены слота выполняется в существующей
  инфраструктуре (вне скоупа клиентского приложения).
- **Напоминания о записи (R-006).** В MVP — системный push за **24 ч и 2 ч** до старта
  (`reminder_hours = [24, 2]`). SMS/WhatsApp/email — Phase 2.

**Технические / платформенные**

- **Mobile-first**: клиентское приложение рассчитано на телефон (использование в зале).
- Клиентское приложение взаимодействует с **существующим бэкендом через API**; слоты, зоны/форматы и
  инструкторы доступны только для чтения.
- **Граница серверной интеграции (R-004).** Детали серверной реализации — транзакционность,
  SLA, маппинг сущностей на внутренние модели бэкенда — **вне скоупа клиентского приложения**.
  Существующий бэкенд — **black-box источник истины**: гарантия «**0 двойных броней**»
  обеспечивается на его стороне (атомарная проверка свободных мест при бронировании). Со стороны
  клиента граница закрывается **контрактом API**: приложение полагается на ответы бэкенда и
  корректно обрабатывает отказ, если мест уже нет.
- **Данные существующей инфраструктуры (R-015).** Это **учебный/тестовый проект**, исторических
  данных и легаси-схемы нет. **Канонической схемой считается контракт API** (OpenAPI в
  `01-analysis/api/`): бэкенд по условию отдаёт ровно те поля, что описаны в контракте, включая
  `rental_price`, `zone_format`, `instructor_info`, `free_rental_equipment`. Поэтому **миграция/backfill
  и стратегия дозаполнения исторических слотов вне скоупа** — отдельных требований к ним не
  формулируем.

**Проект**

- **Срок** — к началу следующего сезона, ~2 месяца (жёсткая цель).
- **Бюджет** ограничен → начинаем с самого нужного, «навороты» позже.

**Границы скоупа**

Скоуп — **клиентское мобильное приложение и API для него**. Управление расписанием, роли
инструктора и владельца, отметка оплаты, оценки инструкторов, админка — **существующая
инфраструктура** или **Phase 2**, вне скоупа текущей поставки.

| В скоупе клиентского приложения | Вне скоупа (существующая инфраструктура / Phase 2) |
| :-- | :-- |
| Регистрация/вход клиента по имени и телефону (SMS-код или аналог) | Роли Инструктора и Владельца/Администратора, админ-экран |
| Просмотр и **фильтрация** списка слотов (дата, тип тренировки, наличие мест, инструктор, **цена**) | Создание и редактирование расписания (слоты приходят из существующего бэкенда через API) |
| Самостоятельная запись **одного места** на слот | Групповая бронь (запись нескольких людей одним аккаунтом) |
| Выбор «своё / прокатное снаряжение», контроль лимита мест и прокатного фонда | Отметка явки/неявки, отмена слота инструктором + массовое уведомление |
| Список своих бронирований | Ручная отметка «оплачено/нет» в админке |
| Отмена записи с правилом 2 часов | Оценка инструктора (звёзды 1–5, без текста) — **Phase 2** |
| Цена и фиксация записи (оплата офлайн) | Онлайн-оплата — **Phase 2** |
| Push-напоминания о записи за 24 ч и 2 ч до старта | Автоматический учёт профилактики по календарю |
| Push при отмене тренировки скалодромом | Программа лояльности для постоянных клиентов — **Phase 2** |
| | Публичный показ рейтинга инструкторов клиентам — **Phase 2** |

## Глоссарий

| Термин | Определение |
| :-- | :-- |
| **Тренировка / слот** | Запланированное групповое занятие: дата/время старта, зона/формат, инструктор, всего и свободно мест, цена. |
| **Зона / Формат** | Вариант тренировки: новичковый (болдеринг, потолок 8) или опытный (трассы, до 16). |
| **Группа** | Состав клиентов, каждый из которых записан отдельной бронью на один слот. |
| **Снаряжение** | Скальники, страховочная система из инвентаря клуба; может выдаваться напрокат. |
| **Аренда / прокат** | Выдача клубного снаряжения клиенту, у которого нет своего; лимит прокатного фонда учитывается отдельно от мест в группе. |
| **Своё снаряжение** | Клиент приходит с собственным снаряжением; занимает место в группе, но не прокатный фонд. |
| **Запись (бронь)** | Резерв **одного места** клиентом на конкретный слот: своё/прокатное снаряжение, статус. Привязана к профилю клиента. |
| **Отмена** | Снятие записи. Клиентом: ранняя (≥2 ч до старта) освобождает место; поздняя (<2 ч) разрешена, фиксируется без штрафов. Скалодромом (организатором / по профилактике): бронь переходит в статус «Отменена скалодромом» с причиной, клиент получает push, повторная запись на этот слот запрещена. |
| **Отмена скалодромом** | Отмена тренировки организатором или из-за профилактики (выполняется в существующей инфраструктуре). На стороне клиента бронь сохраняется со статусом «Отменена скалодромом» + причина; отправляется push-уведомление; повторная запись на отменённый слот недоступна. |
| **Администратор / владелец** | Оля — управляет расписанием, слотами, инструкторами, записями и отметкой оплаты через существующую инфраструктуру (вне скоупа клиентского приложения). |
| **MVP** | Минимально жизнеспособный продукт к старту сезона: расписание, запись одного места, отмены, push-напоминания (24 ч и 2 ч), отображение цены. Оценки, лояльность, онлайн-оплата — Phase 2. |
